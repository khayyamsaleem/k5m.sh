## Architecture Deep Dive

### PicoClaw: The Lightweight Gateway

PicoClaw runs as a single service (the gateway) that:
- **Connects to messaging platforms** (Discord, Telegram, WhatsApp, Signal, etc.)
- **Routes to external APIs** (Groq, Anthropic, OpenAI, local Ollama if available)
- **Executes tools** in a sandboxed subprocess environment
- **Manages context and state** in workspace files

**The dependency chain is minimal:**
```
Discord/Telegram/etc → PicoClaw Gateway → Groq API (or local Ollama)
                                        → Exec sandbox
                                        → File I/O (workspace)
```

**Sandboxing approach:** Process-level isolation. When you execute a command, it runs in a subprocess with:
- Limited filesystem access (workspace only)
- No network access by default
- Dropped Linux capabilities
- Resource limits (memory, CPU, timeout)

**Configuration is straightforward:**
```json
{
  "agents": {
    "main": {
      "model": "groq/openai/gpt-oss-120b",
      "system_prompt": "..."
    }
  },
  "tools": {
    "exec": { "host": "gateway" }
  }
}
```

### OpenClaw: The Full-Featured Stack

OpenClaw runs as a coordinated set of Docker services:

```
┌─────────────────────────────────────────┐
│         Host Machine                    │
│  - Docker daemon                        │
│  - ~/.openclaw/workspace/ (mounted)     │
│  - GPU access (optional)                │
└─────────────────────────────────────────┘
                    │
        ┌───────────┼───────────┬──────────┐
        ▼           ▼           ▼          ▼
    ┌──────┐   ┌──────────┐ ┌────────┐ ┌─────┐
    │Ollama│   │ Gateway  │ │Sandbox │ │ CLI │
    │      │   │ (core)   │ │ (on-   │ │     │
    │      │   │          │ │ demand) │ │     │
    └──────┘   └──────────┘ └────────┘ └─────┘
```

**Each service has a specific job:**
- **Ollama:** Runs local LLMs (qwen, llama, etc.) on GPU
- **Gateway:** The orchestration engine (same as PicoClaw's role)
- **Sandbox:** On-demand Docker containers for isolated execution
- **CLI:** Interactive shell for development and testing

**Sandboxing approach:** Full container isolation. Each task spawns a fresh Docker container that:
- Has its own filesystem (read-only root, writable `/tmp` and `/workspace`)
- Runs as unprivileged user with dropped capabilities
- Has strict resource limits (memory, CPU)
- Dies immediately after task completion
- Can optionally access GPU for profiling

**Configuration is comprehensive:**
```yaml
services:
  ollama:
    image: ollama/ollama:latest
    volumes:
      - ollama-data:/root/.ollama
    environment:
      - OLLAMA_GPU=1

  gateway:
    build: ./gateway
    depends_on:
      - ollama
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
      - ~/.openclaw/workspace:/workspace
```

Plus separate config files for:
- Model routing and fallback chains
- Skill definitions and versioning
- Agent configurations
- System prompts and context

## Operational Complexity: The Real Cost

### PicoClaw Operations

**Getting started:**
1. Install PicoClaw (binary or Docker)
2. Set environment variables (API keys)
3. Edit `config.json` with your model choice
4. Run `picoclaw`
5. Connect Discord bot

**Ongoing maintenance:**
- Rotate API keys occasionally
- Monitor API usage (check Groq/OpenAI dashboards)
- Update PicoClaw binary when new versions release
- Adjust config as needed

**Troubleshooting:** Usually straightforward. Logs are readable, state is visible in workspace files, failures are explicit.

### OpenClaw Operations

**Getting started:**
1. Clone the OpenClaw repo
2. Configure `docker-compose.yml` with GPU settings, volume mounts, resource limits
3. Set environment variables (API keys, Ollama model names)
4. Build custom gateway Docker image (if needed)
5. Run `docker-compose up`
6. Wait for Ollama health check to pass
7. Configure skill marketplace access
8. Connect Discord bot
9. Optional: Set up approval gates for elevated operations

**Ongoing maintenance:**
- Monitor Ollama model downloads and VRAM usage
- Manage Docker volumes (model storage can grow large)
- Handle service interdependencies (if Ollama goes down, gateway waits)
- Orchestrate multi-service restarts
- Debug Docker networking issues
- Update multiple services independently (or together)
- Manage skill versioning and updates

**Troubleshooting:** Requires Docker expertise. Service dependencies create failure modes:
- Ollama health check fails → gateway never starts
- Docker socket mount issues → sandbox can't spawn containers
- Volume permission issues → workspace mount fails
- GPU passthrough misconfiguration → Ollama falls back to CPU

## Sandboxing & Security

Both frameworks sandbox execution, but differently.

### PicoClaw Sandboxing

**Process-level isolation:**
- Commands run in a subprocess with restricted permissions
- Filesystem access limited to workspace directory
- No network access by default
- Linux capabilities dropped (no `CAP_SYS_ADMIN`, etc.)
- Memory and CPU limits enforced by the OS

**Approval model:**
- Optional approval gates for sensitive operations
- You see the command before it runs
- Simple yes/no confirmation

**Threat model:**
- Protects against: AI-generated code that's buggy, poorly written, or mildly malicious
- Doesn't protect against: Sophisticated exploits, kernel vulnerabilities, privilege escalation
- Good enough for: Most use cases (the AI isn't an adversary, it's just not perfect)

### OpenClaw Sandboxing

**Container-level isolation:**
- Each task runs in a fresh Docker container
- Complete filesystem separation (read-only root, writable `/tmp` and `/workspace`)
- Network isolated to localhost
- Runs as unprivileged user with dropped capabilities
- Strict resource limits (memory ceiling, CPU throttle)
- Container destroyed immediately after task completion

**Approval model:**
- Same as PicoClaw: optional approval gates
- More sophisticated: can whitelist safe operations, audit logs

**Threat model:**
- Protects against: Most attacks that don't involve kernel exploits
- Doesn't protect against: Container escape vulnerabilities, host kernel exploits
- Good enough for: Serious use cases (you're comfortable granting filesystem access)

**Verdict:** OpenClaw's container isolation is stronger in theory, but both are "good enough" for practical purposes. The real security comes from the approval gates — you see what's happening before it runs.

## The Hybrid Approach

Here's what I actually do: **Run PicoClaw as the primary daily driver, with OpenClaw as a secondary sandbox for testing and experimentation.**

**Why this works:**
- **PicoClaw handles 95% of tasks** — quick, cheap, reliable
- **OpenClaw tests new skills** — before deploying to production

4. Use PicoClaw for all daily automation

This gives you the best of both worlds: simplicity for production, flexibility for development.

## Final Thoughts: Which One Wins?

**PicoClaw wins on pragmatism.** If you want an AI assistant that works *right now* without fiddling with Docker, PicoClaw is the obvious choice. It's not as flexible, but it's genuinely useful out of the box.

**OpenClaw wins on control.** If you want to understand every layer of your setup and have the sophistication to manage complexity, OpenClaw is worth the effort. The skill system and container isolation are genuinely excellent.

**Neither is "better" — they serve different purposes.** PicoClaw is the daily driver. OpenClaw is the lab.

If I had to recommend one to someone starting out, I'd say: **Start with PicoClaw.** Get it running, use it for a month, understand what you actually need. *Then* evaluate whether OpenClaw's sophistication is worth the operational overhead.

You'll know when you're ready to switch. It's when you find yourself thinking "I wish I could run this locally" or "I need more control over this workflow." That's the moment OpenClaw makes sense.

Until then, PicoClaw is the right choice. Simple, effective, and it just works.
